# CLAUDE.md

## Project Overview

TypeScript monorepo under the `@halfwhey/` npm scope, built with Bun. Currently hosts Zotero-related packages but designed to hold any TypeScript package.

- `packages/zotmeal-core` (`@halfwhey/zotmeal-core`) — Zotero Web API v3 client library, zero runtime dependencies
- `packages/zotmeal-cli` (`@halfwhey/zotmeal-cli`) — CLI using Commander.js
- `apps/mobile` (`@halfwhey/mobile`) — React Native Expo app (**private git submodule**, see `.claude/CLAUDE.local.md`)

## Commands

```bash
bun install                            # Install dependencies
make build                             # Build CLI to packages/zotmeal-cli/dist/zotmeal.js
bun test                               # Run all tests (packages/zotmeal-core/test/)
make typecheck                         # tsc --build + mobile (if submodule present)
bun run scripts/generate-item-types.ts # Regenerate item types from Zotero API schema
bun packages/zotmeal-cli/src/index.ts  # Run CLI from source
```

### Versioning & Publishing

```bash
PACKAGE=zotmeal-cli make version-patch  # Bump a specific package version
PACKAGE=zotmeal-core make version-minor # Works for any package in packages/
make publish                            # Build + publish CLI to npm
```

Only `@halfwhey/zotmeal-cli` is published to npm. It's fully bundled (single file), so its deps are `devDependencies` — npm consumers get no transitive installs.

## Architecture

The core library (`packages/zotmeal-core/src/`) is organized into two modules:

### `api/` — Zotero Web API v3 implementation
```
ZoteroClient (api/client.ts)
  ├── items: ItemsEndpoint        → CRUD + list/search/top/trash/children/export
  ├── collections: CollectionsEndpoint → CRUD + list items/top in collection
  ├── tags: TagsEndpoint          → list, delete
  ├── searches: SearchesEndpoint  → CRUD
  ├── files: FilesEndpoint        → download/view URLs, upload auth, register
  ├── fulltext: FullTextEndpoint  → get/set full-text, list versions
  ├── deleted: DeletedEndpoint    → list deleted since version
  ├── groups: GroupsEndpoint      → get group, list for user
  └── keys: KeysEndpoint         → get current/specific key permissions
      │
      └── HttpClient (api/http.ts)
            → Zotero-API-Key header, Zotero-API-Version: 3
            → Retry on 429 with Retry-After
            → Typed error mapping (403/404/412/429)
            → Conditional GETs (If-Modified-Since-Version / 304)
```

### `utils/` — Utilities operating on API data
- `findDuplicates()` — groups items by matching DOI, ISBN, URL, or title

### Imports
- `@halfwhey/zotmeal-core` — re-exports everything
- `@halfwhey/zotmeal-core/api` — API-only exports
- `@halfwhey/zotmeal-core/utils` — utility-only exports

### Key patterns
- **Pagination**: `paginate()` async generator yields individual items across pages by following `Link: rel="next"` headers. `collectAll()` helper collects into an array.
- **Types**: Item types are a discriminated union on `itemType` (~40 types), generated by `scripts/generate-item-types.ts` from the live Zotero API. Generated files: `packages/zotmeal-core/src/api/types/items.ts`, `packages/zotmeal-core/src/api/types/creators.ts`.
- **Errors**: `ZoteroError` base class with subclasses `AuthenticationError` (403), `NotFoundError` (404), `VersionConflictError` (412), `RateLimitError` (429).
- **CLI**: Commander.js commands → `getClient()` reads env vars → endpoint methods → output formatters (table/json/keys).
- **Config**: Via environment variables (no config file). The CLI reads `ZOTERO_API_KEY`, `ZOTERO_LIBRARY_ID`, and optionally `ZOTERO_LIBRARY_TYPE` (defaults to `"user"`). The core library never reads env vars — it's always the client's responsibility to pass credentials.

## Monorepo Conventions

- **Bun workspaces** (`packages/*`, `apps/*`) handle dependency resolution
- **TypeScript project references**: root `tsconfig.json` references library packages (core, cli). `tsc --build` for incremental compilation. Mobile is excluded from project references (checked separately).
- Library packages export `./src/` directly (Bun-only, no dist build step for libraries)
- New packages: create `packages/<name>/` with `"name": "@halfwhey/<name>"` in `package.json`, add to root `tsconfig.json` references
- Each package has its own `tsconfig.json` extending `../../tsconfig.base.json` with `composite: true`

## Coding Guidelines

- **TypeScript strict mode** is mandatory. The tsconfig uses `exactOptionalPropertyTypes` and `noUncheckedIndexedAccess`.
  - Optional properties must use `prop?: T | undefined` (not just `prop?: T`).
  - Indexed access returns `T | undefined` — always handle the undefined case.
- **Zero runtime dependencies** in `@halfwhey/zotmeal-core`. Use native `fetch`. No axios, got, node-fetch.
- **Use `.js` extensions** in all import paths (TypeScript with bundler module resolution).
- **No unnecessary abstractions**. Keep it simple. Don't add wrappers, helpers, or indirection unless reused in 3+ places.
- **Generated code** (`items.ts`, `creators.ts`) should never be edited manually. Modify `scripts/generate-item-types.ts` and run `bun run scripts/generate-item-types.ts`.
- **Tests** use `bun:test` with mocked `globalThis.fetch`. Tests live in `packages/zotmeal-core/test/`.
- **Auth**: Zotero API uses `Zotero-API-Key` header (not `Authorization: Bearer`).
- When adding new endpoints or types, re-export them from the relevant `index.ts` barrel files.
